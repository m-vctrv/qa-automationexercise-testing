## 2. Цель тестирования

Проверить корректность работы ключевых пользовательских сценариев интернет-магазина (регистрация, авторизация, работа с каталогом, корзиной и оформлением заказа), выявить и задокументировать дефекты функциональности, валидации и интерфейса.

## 3. Объём тестирования (Scope)

### В рамках тестирования (In scope)

1. **Регистрация** — создание нового аккаунта, валидация полей формы.
2. **Авторизация** — вход в аккаунт, выход, обработка неверных данных.
3. **Каталог и поиск** — отображение товаров, поиск, фильтрация по категориям/брендам.
4. **Корзина** — добавление, удаление, изменение количества товаров, подсчёт суммы.
5. **Оформление заказа** — переход к оформлению, заполнение данных, подтверждение.
6. **API** (отдельный слой) — основные эндпоинты сайта: список товаров, поиск, проверка логина, создание/удаление аккаунта.

### Вне рамок тестирования (Out of scope)

- Контактная форма
- Подписка на рассылку (newsletter)
- Раздел Test Cases / API-документация как контент
- Нагрузочное тестирование и тестирование безопасности (отдельные направления)
- Тестирование на мобильных устройствах (только десктоп-браузер)

## 4. Виды тестирования

- **Функциональное** — проверка, что функции работают по ожидаемому поведению.
- **Тестирование интерфейса (UI)** — корректность отображения, вёрстка, элементы.
- **Тестирование валидации** — обработка корректных и некорректных входных данных (позитивные и негативные сценарии).
- **API-тестирование** — проверка ответов сервера, статус-кодов, тела ответа (позитив/негатив).
- **Исследовательское (exploratory)** — поиск дефектов без заранее заданных кейсов.

## 5. Подход к тестированию (Test approach)

- Тестирование проводится методом **чёрного ящика** (через UI) и **серого ящика** (через API и анализ запросов в DevTools).
- При проектировании проверок применяются техники тест-дизайна: **классы эквивалентности, анализ граничных значений**, для форм с несколькими условиями — **таблица решений**, для сценариев с состояниями (например, блокировка/сессия) — **диаграмма состояний и переходов**.
- Для каждого модуля сначала составляется **чек-лист** (высокоуровневые проверки), затем детальные **тест-кейсы** на ключевые и сложные сценарии.
- Найденные дефекты оформляются в виде **баг-репортов** по единому шаблону.
- Локализация дефектов выполняется через **DevTools** (вкладки Network, Console, Elements).

## 7. Критерии входа и выхода

**Критерии начала тестирования (Entry criteria):**
- Доступен сайт и его основной функционал.
- Подготовлены тестовые данные (тестовые аккаунты, данные для форм).
- Готовы чек-листы / тест-кейсы по модулю.

**Критерии завершения тестирования (Exit criteria):**
- Пройдены все запланированные тест-кейсы по модулям из scope.
- Все найденные дефекты задокументированы.
- Результаты сведены в отчёт / репозиторий.

## 9. Риски и ограничения

- Сайт — публичный prod-стенд, окружение может меняться без уведомления (функционал/данные могут обновиться в ходе тестирования).
- Нет доступа к базе данных и серверным логам — проверка ограничена тем, что видно через UI и Network.
- Нет официальных требований (ТЗ) — ожидаемое поведение определяется здравым смыслом и общепринятыми стандартами; спорные места фиксируются как вопросы.
