# API-тестирование: Automation Exercise

Тестирование публичного API интернет-магазина [Automation Exercise](https://automationexercise.com/api_list) с помощью Postman.

## Объект тестирования

Публичные REST-эндпоинты сайта Automation Exercise. Проверяются позитивные и негативные сценарии, корректность статус-кодов, структуры и содержимого ответов.

## Инструменты

- **Postman** — отправка запросов и автоматические проверки (tests)
- Формат коллекции: Collection v2.1

## Структура коллекции

Коллекция `Automation Exercise API` содержит 6 запросов, сгруппированных в две папки:

### Products
| Запрос | Метод | Эндпоинт | Сценарий |
|---|---|---|---|
| Get Products List | GET | `/api/productsList` | Позитивный: получение списка товаров |
| Search Product - valid | POST | `/api/searchProduct` | Позитивный: поиск по валидному запросу |
| Search Product - missing param | POST | `/api/searchProduct` | Негативный: запрос без обязательного параметра |

### Auth
| Запрос | Метод | Эндпоинт | Сценарий |
|---|---|---|---|
| Verify Login - valid | POST | `/api/verifyLogin` | Позитивный: вход с верными данными |
| Verify Login - wrong data | POST | `/api/verifyLogin` | Негативный: неверные email/пароль |
| Verify Login - missing param | POST | `/api/verifyLogin` | Негативный: запрос без обязательного параметра |

## Покрытые проверки (tests)

В каждом запросе настроены автоматические проверки:
- HTTP статус-код ответа
- `responseCode` в теле ответа
- содержимое сообщения (`message`)
- наличие и непустота данных (для списка товаров)
- время ответа

## Обнаруженный дефект

При тестировании выявлено **системное несоответствие HTTP статус-кода фактическому результату**:

- API всегда возвращает **HTTP-статус 200**, даже при ошибках;
- реальный код результата передаётся только в теле ответа в поле `responseCode` (400, 404 и т.д.).

Например, запрос без обязательного параметра возвращает HTTP 200, но в теле:
```json
{
    "responseCode": 400,
    "message": "Bad request, ... parameter is missing in POST request."
}
```

Это нарушает семантику REST (статус-код должен отражать результат запроса) и может приводить к тому, что клиенты и системы мониторинга, ориентирующиеся на HTTP-код, не распознают ошибку. Дефект воспроизводится на нескольких эндпоинтах, что подтверждает его системный характер.

> В автотестах фактическое поведение (HTTP 200) зафиксировано как проходящая проверка, а несоответствие явно отмечено в названии теста как известный дефект.

## Как запустить

1. Установить [Postman](https://www.postman.com/downloads/).
2. Импортировать файл `Automation Exercise API.postman_collection.json` (Import → выбрать файл).
3. Для запросов `Verify Login` подставить валидные email и пароль зарегистрированного аккаунта.
4. Запустить запросы по отдельности или всю коллекцию через Collection Runner — результаты проверок отобразятся во вкладке Test Results.
